# ALamal-AB · Obada — متطلبات مشروع الجملة (Wholesale)

> **الغرض:** توثيق حرفي لطلبات المدير الجديد، وما يبقى كما هو من CLOTEX، وأين تُعدَّل الواجهة والمنطق.  
> **الحالة:** جاهز للبدء بعد نسخ المشروع إلى مجلد `Obada`.  
> **آخر تحديث:** 2026-06-12  
> **اسم المشروع:** **ALamal-AB Obada**

---

## 1. ملخص تنفيذي

مشروع **Obada** نسخة مخصصة من نظام الأقمشة الحالي (CLOTEX)، لعميل **بائع جملة**:

- يشتري آلاف الأتواب من الصين (ملف Excel تفصيلي لكل توب).
- يبيع **بالعدد (توب)** وليس بالمتر أمام الزبون.
- يحتاج **تفنيد** (طول كل توب) عند التسليم من المستودع.
- يحتاج قسم **التسليم** بين فاتورة البيع وتنفيذ أمين المستودع.

**كل ما عدا دورة الشراء / البيع / التسليم يبقى متشابهاً جداً:**

| يبقى كما هو (نفس CLOTEX) | يتغيّر في Obada |
|--------------------------|-----------------|
| كشف حساب عملاء | استيراد شراء Excel صيني |
| كشف حساب موردين | تسعير متر/ياردة عند الاستيراد |
| الصناديق والسندات | فاتورة بيع **بعدد الأتواب** |
| التقارير (عامة) | قسم **التسليم** |
| المحاسبة الأساسية | **التفنيد** داخل التسليم |
| التصنيفات (مع تغذية تلقائية من الاستيراد) | باركود نظام + مرجع مورد (بدون إخفاء) |

## 2. فهم دورة الشراء (من الصين)

### 2.1 السيناريو

1. المورد الصيني يرسل ملف Excel (مثال واقعي: `Roll List 336 DENIM.xls`).
2. الملف يحتوي **قائمة تعبئة تفصيلية** وليس سعراً فقط:
   - اسم الخامة (مثال: `Amelia-19 66/67` — DENIM).
   - إجمالي: عدد الأتواب + إجمالي الأمتار.
   - لكل توب: **رقم التوب** + **الطول (متر أو ياردة)** + **اللوط**.
   - أطوال الأتواب **غير موحّدة** (مثال: 475 توب، 37,940 م، أطوال من 34م إلى 111م).
3. عند الاستيراد في النظام:
   - يُحلَّل الملف ويُعرض للمستخدم ما فيه (عدد أتواب، أطوال، لوطات، خامة…).
   - يُسعَّر **المتر أو الياردة** أثناء الاستيراد (حسب نظام/وحدة الملف).
   - يُضاف **تلقائياً** إلى كل الأقسام التي تحتاج البيانات:
     - **المخزون** (أتواب + أطوال).
     - **التصنيفات** (اسم خامة، نوع، ألوان… حسب ما يوفره الملف).
     - **المالية/المحاسبة** (فاتورة شراء، تكلفة، ذمة مورد).

### 2.2 الباركود عند الشراء

- **لا يوجد باركود من بلد المنشأ (الصين)** حالياً.
- النظام يولّد **باركود خدمة / نظام** لكل توب.
- **حقل مرجع المورد / رقم التوب الأصلي يبقى ظاهراً** — لا يُخفى.
- السبب: الزبون قد يطوّر لاحقاً البيع والشراء ويضيف باركود حقيقي؛ البنية يجب أن تدعم ذلك دون إعادة بناء.

### 2.3 مرجع ملف Excel (مثال محلل)

```
DETAILED PACKING LIST
الخامة: Amelia-19 66/67
الإجمالي: 475 ROLLS · 37,940 M
أعمدة متوازية (3 أتواب في كل صف):
  ROLL NO. | LENGTH (M) | LOT
```

## 3. فهم دورة البيع (جملة)

### 3.1 السيناريو اليومي

1. زبون يطلب مثلاً: **«20 توب قماش Galaxy»**.
2. **بغض النظر** عن العرض أو الطول عند اتفاق البيع — الوحدة التجارية = **عدد الأتواب**.
3. تُنشأ **فاتورة بيع** بالعدد (20 توب)، مع الألوان وأكواد الألوان وكود الخامة كما في النظام الحالي.
4. الفاتورة **لا تُغلق مباشرة** — تنتقل إلى **قسم التسليم** (انظر §4).

### 3.2 التفنيد (Tafnid)

- زر/عملية اسمها **تفنيد**.
- بعد البيع بالعدد، يجب تحديد **طول كل توب مباع** لأن الطول غير موحّد.
- الوحدة: **متر أو ياردة** حسب التوب.
- مثال: بيع 50 توب Galaxy → نافذة تفنيد: التوب 1 = 79م، التوب 2 = 85م، …

### 3.3 الفرق عن CLOTEX الحالي

| CLOTEX | Obada |
|--------|-------|
| بيع غالباً بالمتر/الطول من الثوب | بيع **بعدد الأتواب** في سطر الفاتورة |
| خصم مخزون مرتبط بطول البيع مباشرة | خصم نهائي بعد **التسليم + التفنيد** |
| باركود من البداية غالباً | باركود نظام + رقم توب مورد |

## 4. قسم التسليم (طلب حرفي من المدير)

### 4.1 المسار

```
إنشاء فاتورة بيع جديدة (بالعدد: X توب)
        ↓
الفاتورة تذهب إلى قسم «التسليم» (ليست منتهية)
        ↓
أمين المستودع يستلم الطلب في هذا القسم
        ↓
طباعة إيصال / ورقة استلام للمستودع
        ↓
أمين المستودع يُفنّد محتويات الفاتورة (أطوال كل توب)
        ↓
تأكيد التسليم → خصم المخزون → إغلاق/إكمال دورة البيع
```

### 4.2 الأدوار

| الدور | المهمة |
|-------|--------|
| **المبيعات / المدير** | إنشاء فاتورة بيع بعدد الأتواب |
| **أمين المستودع** | استلام من قسم التسليم، تفنيد، تسليم فعلي، طباعة إيصال |

### 4.3 حالة الفاتورة المقترحة

```
DRAFT → CONFIRMED_SALE → IN_DELIVERY → FULFILLED (بعد التفنيد)
```

(التسميات النهائية تُثبَّت عند التنفيذ.)

## 5. ما يبقى دون تغيير جوهري

- كشف حساب **عملاء** — نفس المبدأ.
- كشف حساب **موردين** — نفس المبدأ.
- **الصناديق** و**السندات** (قبض/صرف).
- **التقارير** العامة — نفس الفكرة (قد تُضاف تقارير جملة لاحقاً).
- شجرة **التصنيفات** — نفس البنية؛ مع **إنشاء/تحديث تلقائي من استيراد الصين**.
- المستخدمون والصلاحيات — نفس النموذج (مع صلاحية قسم التسليم لأمين المستودع).

## 9. قرار المعمارية: نسخ المشروع إلى `Obada`

**موافق — هذا هو المسار الصحيح.**

```
CLOTEX (الحالي)          Obada / ALamal-AB (جديد)
     │                            │
     │                            ├─ نفس: كشوف، صناديق، سندات، تقارير
     │                            └─ مختلف: شراء Excel صيني، بيع بالتوب، تسليم، تفنيد
     │
     └─ لا تعديل يكسر العميل الحالي
```

**بعد نسخك للمجلد:**

1. أخبرنا بمسار المجلد الجديد.
2. نبدأ بالمرحلة 1 ثم 2 (مستورد `Roll List 336 DENIM.xls` كنموذج أول).

## 10. أسئلة مفتوحة (للتثبيت مع المدير لاحقاً)

1. هل التفنيد **إجباري** قبل إغلاق أي فاتورة في التسليم؟
2. عند التفنيد: هل يختار أمين المستودع **أرقام أتواب محددة** من المخزون أم يدخل أطوالاً فقط؟
3. سعر البيع للزبون: على **التوب** أم على **المتر** بعد التفنيد؟
4. هل فاتورة السعر من الصين دائماً في ملف منفصل عن قائمة الأطوال؟
