# تقرير: ضبط `DATABASE_URL` لنفق SSH (127.0.0.1:5433) — CLOTEX

**التاريخ:** 2026-05-03  
**السرية:** لا يحتوي هذا التقرير على كلمات مرور، ولا `DATABASE_URL` كاملاً، ولا مفاتيح أو توكنات.

---

## 1. مصدر بيانات الاعتماد

| المصدر | النتيجة |
|--------|---------|
| **`VPS.md`** | لا يُخزّن كلمات مرور؛ يوجّه إلى `server/.env` و`VPS.example.md`. |
| **`server/.env` (محلي، غير مُرفوع)** | يحتوي مسبقاً على مستخدم `erp_user` وقاعدة `fabric_erp` وكلمة مرور PostgreSQL مُعرَّفة محلياً — **لم يُستخرَج من الوثائق العامة**. |
| **استنتاج** | كلمة مرور `erp_user` **موجودة بالفعل في بيئة العمل المحلية** في `server/.env`؛ لم تكن مطلوباً من مستودع Git. |

---

## 2. تحديث `server/.env`

| البند | الحالة |
|--------|--------|
| تم ضبط الهدف على **`127.0.0.1:5433`** (منفذ النفق المحلي) بدلاً من `5432`. |
| بقي دون تغيير | `ACTIVATION_KEY_PEPPER`, `JWT_SECRET`, `SEED_ADMIN_PASSWORD`, `ACTIVATION_REQUIRE_ACTIVE`, وباقي المتغيرات كما طُلب. |
| الصيغة المنطقية | `postgresql://erp_user:***@127.0.0.1:5433/fabric_erp` مع معاملات إضافية حسب الملف (مثل وضع SSL كما كان). |

---

## 3. التحقق من النفق على Windows

| الفحص | النتيجة |
|--------|---------|
| `netstat -ano \| findstr :5433` | **LISTENING** على `127.0.0.1:5433` و`[::1]:5433`. |
| PID المرتبط | العملية **`ssh.exe`** (وليس `postgres.exe`) — يتوافق مع نموذج: عميل SSH يفتح النفق محلياً. |

---

## 4. اختبار اتصال قاعدة البيانات

- **الأداة:** سكربت محلي `server/src/scripts/tunnelDbPing.ts` (يقرأ `server/.env` فقط؛ لا يطبع أسراراً).
- **النتيجة (حقول آمنة فقط):**

```json
{
  "db": "fabric_erp",
  "u": "erp_user",
  "server_port": "5432"
}
```

| الحقل | المعنى |
|--------|--------|
| `db` | القاعدة المتصلة = **`fabric_erp`** |
| `u` | المستخدم = **`erp_user`** |
| `server_port` | من جهة **خادم PostgreSQL على VPS** يظهر غالباً **`5432`** — هذا متوقع؛ النفق يوجّه **الاتصال المحلي 5433 → 5432 على الجانب البعيد**. |

**خلاصة:** الاتصال عبر النفق **نجح** بعد تحديث المنفذ إلى **5433**.

## 7. إذا لم تكن كلمة المرور موجودة لاحقاً

لا تُخمَّن. خيارات آمنة:

- **A.** على VPS: إعادة تعيين كلمة مرور دور `erp_user` عبر `sudo -u postgres psql` (تنفيذ يدوي من المالك).
- **B.** إنشاء دور/كلمة مرور جديدة ومنح الصلاحيات، ثم تحديث `server/.env` محلياً فقط.
- **C.** الاسترجاع من ملاحظات خاصة غير مُرفوعة (خارج Git).

---

## 8. أوامر مقترحة للخطوة التالية (بدون تنفيذ من هذا التقرير)

بعد التأكد من النسخ الاحتياطي على VPS:

1. `pg_dump` (حسب دليل النسخ الاحتياطي في تقرير VPS).  
2. `npm run server:reset:clean` (مع المتغيرات المطلوبة وبـ `DATABASE_URL` الصحيح).  
3. `npm run activation:generate` (مع `ACTIVATION_KEY_PEPPER` المضبوط).

إعادة فحص الاتصال بعد أي تغيير:

```bash
npx tsx server/src/scripts/tunnelDbPing.ts
```
