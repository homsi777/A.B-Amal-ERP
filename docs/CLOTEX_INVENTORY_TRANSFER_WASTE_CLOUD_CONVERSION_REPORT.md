# تقرير تحويل المناقلة والتوالف إلى كيانات سحابية — CLOTEX ERP

**التاريخ:** 2026-05-03

---

## 1. الملخص

تم استبدال البيانات الوهمية في صفحتي **المناقلة بين المستودعات** و**إهلاك مادي / التوالف** بطبقة كاملة من **PostgreSQL + REST API** مع الحفاظ على التخطيط البصري العام (شبكة عمودين للمناقلة، جداول بـ `bg-slate-800`، بطاقات إحصائية للتوالف، RTL). التأكيد على مستوى الخادم يحدّث **مواقع الأتواب** في `fabric_rolls`، ويسجّل **حركات مخزون** (`inventory_movements`) مع مراجع `INVENTORY_TRANSFER` و`INVENTORY_WASTE`، مع توافق مع أنواع الحركة الحالية (`TRANSFER_OUT` / `TRANSFER_IN`، `DAMAGE`).

---

## 2. مصادر الوهم السابق التي أُزيلت

| الصفحة | البيانات الوهمية السابقة |
|--------|---------------------------|
| `Transfers.tsx` | مصفوفة `useState` تحتوي TRN-1092، مستودعات ثابتة، حقول غير موصولة |
| `Depreciation.tsx` | مصفوفة SPO-881، مجموع 165$ ثابت، أسماء مستخدمين وهمية |

لا توجد صفوف تجريبية في البذرة لهذه الوحدات.

تم تنفيذها بنجاح على بيئة التطوير (`npm run server:migrate`).

## 4. الجداول المُنشأة

| الجدول | الغرض |
|--------|--------|
| `inventory_transfers` | رأس طلب المناقلة (مسودة / مؤكد / ملغى) |
| `inventory_transfer_lines` | أثواب لكل طلب، مع قيد فريد لكل `(transfer_id, fabric_roll_id)` لكل شركة |
| `inventory_waste_records` | رأس سجل التوالف |
| `inventory_waste_lines` | أثواب لكل سجل، مع حقول طول/وزن مهلك اختيارية |

فهارس على `company_id`، التواريخ، الحالة، والحقوم المرجعية حسب المواصفات.

---

## 5. مسارات الخادم

| المسار | الملف |
|--------|--------|
| `/api/inventory/transfers` | `server/src/routes/inventoryTransferRoutes.ts` |
| `/api/inventory/waste` | `server/src/routes/inventoryWasteRoutes.ts` |

التسجيل في `server/src/app.ts`. جميع النهايات محمية بـ **`authenticateRequest`** وتصفية **`company_id`** من JWT.

**المناقلة:** قائمة، تفاصيل، إنشاء، تحديث (مسودة)، تأكيد، إلغاء (مسودة فقط). رقم المناقلة يُولَّد بالخادم (`generateDocumentNo('TRF')`).

**التوالف:** نفس النمط مع رقم `WST`. التأكيد يسجل حركة `DAMAGE`؛ التوالف الكامل يضع حالة الثوب `DAMAGED`؛ الجزئي يخفض `length_m` ويحدّث الوزن المحسوب عند الإمكان.

## 7. تحويل شاشة المناقلة

- تحميل المناقلات من API مع بحث.
- نموذج يسار: مستودع/موقع مصدر ووجهة، إضافة أثواب من المستودع المصدر (استعلام `listFabricRolls`)، ملاحظات، **حفظ مسودة**.
- جدول يمين: رقم الطلب، الأسماء، عدد الطاقات، شارات الحالة، **تأكيد / إلغاء** للمسودات فقط.
- حالة فارغة: **«لا توجد مناقلات بين المستودعات بعد»**.

---

## 8. تحويل شاشة التوالف / الإهلاك

- إزالة القيمة النقدية الوهمية؛ بطاقتان: إجمالي السجلات من الخادم، وعدد المسودات (استعلام ثانٍ بـ `status=DRAFT`).
- لوحة إنشاء مسودة قابلة للطي: نوع التوالف، مستودع (مطلوب لتجميع الأثواب)، موقع، سبب، ملاحظات، إضافة أثواب مع طول مهلك اختياري للتوالف الجزئي.
- جدول السجل مع أعمدة متوافقة مع البيانات الحقيقية (بدون عمود مستخدم إن لم يُعرَض من الخادم في القائمة).
- حالة فارغة: **«لا توجد توالف أو إهلاكات مسجلة بعد»**.

---

## 9. تكامل سجل حركة الثوب

- حركات المناقلة تُدرج كـ `TRANSFER_OUT` أو `TRANSFER_IN` مع `reference_type` و`reference_no`، والملاحظة «مناقلة بين المستودعات».
- التوالف يُدرج كـ `DAMAGE` مع الملاحظة «توالف / إهلاك مادة».
- تم تحديث تسميات أنواع الحركة في `RollDetails.tsx`: **«مناقلة (خارج/داخل)»** لسهولة القراءة العربية.

## 13. قيود معروفة (MVP)

- إلغاء **المناقلة أو التوالف المؤكدة** غير مدعوم؛ يُرفض من الخادم مع رسالة توضيحية.
- المناقلة: نقل **ثوب كامل فقط** لكل سطر (`quantity = 1`).
- لا عكس تلقائي لحركات المخزون عند إلغاء مسودة (لا حركات تُنشأ للمسودات).
- التوالف الجزئي يخفض الطول؛ لا محرك محاسبة خسائر.
