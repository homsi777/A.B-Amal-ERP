# CLOTEX — سير العمل الرسمي: تطبيق Electron محلي على Windows + PostgreSQL على VPS عبر نفق SSH

**التاريخ:** 2026-05-03  
**السرية:** لا يتضمن هذا الملف كلمات مرور، ولا `DATABASE_URL` كاملاً، ولا مفاتيح SSH أو JWT.

---

## 1. الملخص

- المشروع **يبقى بالكامل على جهاز Windows** المحلي؛ **لا** يُرفع المشروع إلى الـVPS **ولا** تُشغَّل أوامر `npm` على الـVPS لاستخدام واجهة التطوير.
- **PostgreSQL** يعمل على **الـVPS** ويستمع داخلياً عادة على `127.0.0.1:5432` (وليس للإنترنت مباشرة).
- الاتصال من Windows يتم بـ **نفق SSH**: المنفذ المحلي **`127.0.0.1:5433`** على Windows يُوجَّه إلى **`127.0.0.1:5432`** على الـVPS (حيث يعمل Postgres).
- الواجهة الخلفية والـ API تعملان **محلياً** على الجهاز؛ المتغير **`DATABASE_URL`** في `server/.env` يشير إلى **`127.0.0.1:5433`** (وليس إلى عنوان عام للـVPS لـPostgres).

---

## 2. البنية الصحيحة (مخطط نصي)

```
[ Electron + Vite ] على Windows
        ↓ HTTP محلي
[ Backend / Fastify ] على Windows (مثلاً المنفذ 4010)
        ↓ PostgreSQL عبر libpq
[ 127.0.0.1:5433 ] على Windows  ←── هذا هو مدخل النفق المحلي فقط
        ↓ SSH tunnel (encrypted)
[ VPS: SSH يصل إلى 127.0.0.1:5432 حيث PostgreSQL ]
```

- **الـVPS هنا دوره استضافة قاعدة البيانات فقط** في هذا السيناريو، وليس تشغيل استنساخ المشروع أو `npm run electron:dev` على الخادم.

---

## 3. أمر النفق اليدوي (المعتمد رسمياً)

في **نافذة طرفية منفصلة** على Windows، مع إبقاء النافذة مفتوحة أثناء العمل:

```bash
ssh -p 2727 -L 5433:127.0.0.1:5432 ubuntu@65.21.136.217
```

- **المستخدم:** `ubuntu` (ليس `root` افتراضياً لمطابقة الإعداد العامل لديك).
- **منفذ SSH:** `2727`.
- **كلمة مرور SSH:** تُدخل تفاعلياً عند الطلب (مسموح ومتوافق مع هذا السير العمل).

استبدل عنوان الـIP إذا كان خادمك مختلفاً.

---

## 4. لماذا المنفذ المحلي 5433 وليس 5432 على Windows؟

| المنفذ | المعنى |
|--------|--------|
| **5433 على Windows** | نقطة الالتقاء التي يفتحها عميل SSH محلياً؛ التطبيق والـ backend يتصلان بـ Postgres «كما لو» كان محلياً هنا. |
| **5432 على الـVPS** | المنفذ الذي يستمع عليه PostgreSQL **داخل الخادم** بعد أن يمر الاتصال عبر النفق. |
| **5432 على Windows** | غالباً ما يكون إما Postgres محلي على الجهاز أو التباساً؛ **لا يُفترض** أن يكون هو منفذ اتصالك بقاعدة الـVPS في هذا النموذج. استخدم **`127.0.0.1:5433`** في `DATABASE_URL`. |

---

## 5. لماذا يظهر `server_port: "5432"` في نتيجة `tunnelDbPing` وهذا طبيعي؟

الاستعلام يستخدم `inet_server_port()` وهو **منفذ خادم PostgreSQL الفعلي على الآلة التي تعمل عليها قاعدة البيانات** (على الـVPS داخلياً `5432`).  
الاتصال من التطبيق يمر أولاً إلى **`127.0.0.1:5433`** على Windows؛ بعد النفق، الخادم يظل يعرّف نفسه على المنفذ **5432**. لا تعارض بين ذلك وبين استخدام **5433** في `DATABASE_URL` على جهازك.

---

## 6. صيغة `DATABASE_URL` في `server/.env` (بدون كشف كلمة مرور)

يجب أن يكون الهدف **`127.0.0.1`** والمنفذ **`5433`** (مع النفق النشط):

```env
DATABASE_URL=postgresql://erp_user:<password>@127.0.0.1:5433/fabric_erp?sslmode=disable
```

- ضع كلمة مرور مستخدم قاعدة البيانات `<password>` في الملف الحقيقي `server/.env` فقط محلياً ولا ترفعه إلى Git.

**التحقق من الملف المحلي:** تم التحقق أن الإعداد يستخدم **`127.0.0.1:5433`** في بيئة التطوير (وليس تعريفاً إلى منفذ 5432 على Windows لهذا الغرض).

---

## 7. الترتيب الصحيح للتشغيل أثناء التطوير

### أ) وضع Windows — أمر واحد (موصى به)

من جذر المشروع:

```powershell
npm run electron:dev
```

السلوك: إذا لم يكن المنفذ **`127.0.0.1:5433`** مفتوحاً، يفتح السكربت **نافذة SSH جديدة** لإدخال كلمة مرور الخادم؛ بعد جاهزية النفق يشغّل فحص القاعدة ثم **السيرفر + Vite + Electron**. عنوان الـVPS الافتراضي للعرض هو القيمة في السكربت ويمكن تجاوزها بـ **`FABRIC_VPS_HOST`**، ومستخدم SSH بـ **`FABRIC_VPS_SSH_USER`** (افتراضي `ubuntu`). إذا كان النفق **مفتوحاً مسبقاً**، يتخطى فتح SSH ويذهب مباشرة للفحص ثم التطبيق.

لا يزال بالإمكان تشغيل النواة فقط بدون مسار النفق التلقائي:

```powershell
npm run electron:dev:stack
```

(مفيد على أنظمة لا تستخدم PowerShell، أو عند تشغيل النفق يدوياً مسبقاً.)

### ب) الطريقة اليدوية (كل أنظمة التشغيل أو إذا فضّلت طرفية منفصلة)

1. فتح **نفق SSH** في طرفية وتثبيتها مفتوحة (يدويّاً بالأمر أعلاه).
2. التأكد أن المنفذ المحلي يستمع (انظر القسم التالي).
3. التحقق من اتصال القاعدة (اختياري لكن موصى به): `npm run db:tunnel:check` أو `npx tsx server/src/scripts/tunnelDbPing.ts`.
4. تشغيل الأوامر المحلية من **جذر المشروع** حسب الحاجة، مثل:
   - `npm run server:migrate`
   - `npm run server:seed`
   - `npm run server:check`
   - `npm run build`
   - **`npm run electron:dev`** على Windows كما في (أ)، أو **`npm run electron:dev:stack`** بعد فتح النفق يدوياً.

## 8. التحقق من النفق

```powershell
netstat -ano | findstr :5433
```

توقّع حالة **LISTENING** على عنوان محلي يتعلق بالمنفذ **5433** عندما يكون النفق يعمل.

---

## 9. التحقق من اتصال قاعدة البيانات (بدون طباعة أسرار)

```powershell
npx tsx server/src/scripts/tunnelDbPing.ts
```

أو:

```powershell
npm run db:tunnel:check
```

المخرجات الآمنة المتوقعة (حقول فقط):

```json
{
  "db": "fabric_erp",
  "u": "erp_user",
  "server_port": "5432"
}
```

السكربت **لا** يطبع `DATABASE_URL` أو كلمات مرور أو `JWT_SECRET` أو فلفرة التفعيل.

---

## 10. ما الذي لا يجب فعله

- **لا** افترض مستخدم SSH `root` كافتراض مشروعنا الرسمي؛ المعتمد في التوثيق **`ubuntu`** مع المنفذ **2727** كما في النفق اليدوي.
- **لا** تجبر استخدام مفتاح SSH إذا اعتمدت كلمة المرور؛ النفق اليدوي التفاعلي هو المرجع.
- **لا** ترفع المشروع إلى الـVPS لهذا سير العمل.
- **لا** تشغّل `npm run electron:dev` على الـVPS لهذا النموذج.
- **لا** تعتمد على تعريض PostgreSQL على الإنترنت العام (منفذ 5432 العام) إذا كان السياسة الأمنية تعتمد النفق فقط.

---

## 11. أوامر مساعدة

### `npm run electron:dev:vps` (اختياري)

- السكربت **`scripts/electron-dev-vps.ps1`** **لا يشغّل SSH**: إذا لم يكن هناك مستمع على `127.0.0.1:5433` يطبع أمر النفق اليدوي ويخرج؛ إذا كان النفق جاهزاً ينفّذ **`db:tunnel:check`** ثم **`npm run electron:dev:stack`**.
- للسير الكامل من نقطة الصفر على Windows يُفضَّل **`npm run electron:dev`** (يفتح نافذة SSH عند الحاجة).

### `npm run electron:dev` مقابل `electron:dev:stack`

- **`electron:dev`**: على Windows يمر عبر **`scripts/electron-dev-all.ps1`** (نفق تفاعلي عند الحاجة + فحص قاعدة + السلسلة الكاملة).
- **`electron:dev:stack`**: تجميع Electron فقط (compile + concurrently) بدون التحقق من النفق — استخدمه بعد نفق يدوي أو على macOS/Linux حيث لا يُستدعى السكربت تلقائياً بنفس الطريقة (يمكن تشغيل النفق يدوياً ثم **`electron:dev:stack`**).

---

## 12. لماذا كان سلوك السكربت السابق مضللاً؟

كان السكربت السابق يحاول بدء `ssh` في الخلفية مع **`BatchMode=yes`** وهدف مستخدم افتراضي **`root`** واعتماد ضمني على **مفتاح SSH**؛ ذلك لا يطابق سير العمل الفعلي لديك:

- الدخول التفاعلي بكلمة المرور تحت **`ubuntu`** على المنفذ **2727**.
- عدم وجوب وجود مفتاح محلي.

تم استبدال ذلك بفحص المنفذ المحلي وإرشاد صريح إلى النفق اليدوي والاحتفاظ بـ **`npm run electron:dev`** كمسار التطبيق الرئيسي.

## 13. الملفات والسكربتات التي تغيّرت

| الملف | التغيير |
|--------|---------|
| `scripts/electron-dev-all.ps1` | مطابقة **`npm run electron:dev`**: نافذة SSH عند غياب النفق، انتظار **5433**، ثم **`db:tunnel:check`** ثم **`electron:dev:stack`**. |
| `scripts/electron-dev-vps.ps1` | التحقق من النفق ثم **`electron:dev:stack`** (لا استدعاء متسلسل لـ `electron:dev`). |
| `package.json` | **`electron:dev`** يستدعي السكربت أعلاه؛ **`electron:dev:stack`** يحمل سلسلة التجميع + concurrently الأصلية. |
| `server/src/scripts/tunnelDbPing.ts` | `dotenv` مع **`quiet: true`** لتقليل الضوضاء في الطرفية. |
| `server/.env.example` | تعليق عربي وصيغة مثال لـ `DATABASE_URL` على **127.0.0.1:5433**. |
| هذا المستند | تحديث سير العمل عند الحاجة. |

## 15. الخلاصة النهائية

تم توحيد طريقة التشغيل الرسمية: التطبيق محلي على Windows ويتصل بقاعدة PostgreSQL على VPS عبر نفق SSH على المنفذ المحلي **5433**، مع تركيز أوامر التطوير على **`npm run electron:dev`** بعد فتح النفق يدوياً والإبقاء على نافذة SSH مفتوحة.
