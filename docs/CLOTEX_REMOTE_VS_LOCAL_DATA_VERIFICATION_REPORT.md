# تقرير التحقق الفني: بيانات CLOTEX — PostgreSQL البعيد مقابل التخزين المحلي

**التاريخ:** 2026-05-03  
**الطبيعة:** تحقق قائم على مراجعة الشيفرة وبنية التشغيل (بدون كشف أسرار).  
**ملاحظة:** استعلامات `psql` المباشرة على VPS تُنفَّذ لدى المالك حيث يتوفر `DATABASE_URL`؛ التقرير يثبت **أين** تُخزَّن البيانات منطقياً، و**كيف** يُتحقق ميدانياً.

---

## 1. ملخص تنفيذي

- **الخادم (Node.js / Fastify):** يتصل **فقط** بقاعدة **PostgreSQL** عبر `pg` و`DATABASE_URL` من `server/.env`. لا يوجد في مسار تشغيل المشروع **SQLite** ولا **قاعدة أعمال محلية** داخل الحزمة.
- **التفعيل:** مفاتيح التجزئة، الأحداث، حالة التفعيل في `system_settings`، والأجهزة (حيث تُستخدم) تُسجَّل عبر **SQL في PostgreSQL** — ليست «مصدر حقيقة» في المتصفح أو Electron.
- **المستخدمون وتسجيل الدخول:** التحقق من كلمة المرور عبر **bcrypt** مقابل `password_hash` في جدول **`users`** على PostgreSQL؛ العميل يحتفظ بـ **رمز JWT** محلياً فقط (جلسة).
- **بيانات الأعمال (عملاء، موردون، أتواب، مستودعات، إلخ):** المسارات الحديثة في `server/src/routes/*` تستخدم `getPool()` → PostgreSQL. الواجهة تقرأ/تكتب عبر **REST API**.
- **محلياً (مقبول):** تفضيلات واجهة، مسار API اختياري، رمز جلسة، إعدادات سطح مكتب/طابعات، مسودة إعدادات — **ليس** مصدر حقيقة لبيانات الأعمال.
- **حالة «سحابي عملياً»:** إذا كان `VITE_API_BASE_URL` (أو إعداد سطح المكتب) يشير إلى خادم API يعمل على VPS و`DATABASE_URL` على ذلك الخادم يشير إلى **PostgreSQL على نفس الـ VPS أو خدمة مُدارة**، فالمصدر الحقيقي للبيانات هو **قاعدة البعيد**. التطوير المحلي (`127.0.0.1`) يبقى PostgreSQL وليس تخزيناً وهمياً، لكنه **ليس** VPS حتى تغيّر الاتصال.

**الحكم المختصر:** **المشروع يعتمد على PostgreSQL (عنوانه من `DATABASE_URL` على خادم الـ API) كمصدر حقيقة للتفعيل والمستخدمين وبيانات الأعمال**؛ التخزين المحلي للعميل يقتصر على الجلسة والإعدادات.  
إذا وُجدت وحدات واجهة قديمة تستخدم `useStore` (Zustand) بذاكرة فقط، فهي **ذاكرة جلسة / واجهة** وليست قاعدة بيانات منفصلة — إعادة التحميل دون API لا تسترجع بيانات الأعمال من الخادم.

## 4. إثبات هدف اتصال الخادم بقاعدة البيانات

### تحميل البيئة

- `server/src/config/env.ts` يحمّل `server/.env` ويتطلب **`DATABASE_URL`** غير فارغ (Zod).

### إنشاء العميل

- `server/src/db/pool.ts`: يُنشئ `pg.Pool` من **`connectionString` مشتق من `DATABASE_URL`** فقط (مع `client_encoding=UTF8`). لا يوجد مسار بديل لـ SQLite أو ملف JSON كقاعدة.

### ماذا يُذكر في التقرير بأمان

| البند | القيمة المعروضة |
|--------|-----------------|
| محرك DB | **PostgreSQL** |
| مصدر الاتصال | **`server/.env` → `DATABASE_URL`** |
| الهدف الزمني | **الخادم الذي تصل إليه سلسلة الاتصال** (مثال: `postgresql://user@***:***/dbname` — **لا نلصق السلسلة كاملة في التقرير**) |
| التحقق الصحي | على الخادم يمكن تنفيذ (بدون كشف كلمات سر): `SELECT current_database(), current_user, version();` ومراجعة أن الهوست في `DATABASE_URL` يطابق VPS المتوقع (أول جزء من الهوست بعد `@` يمكن إخفاؤه: `db.***.***`). |

**خلاصة:** وقت تشغيل الـ API، **كل** قراءة/كتابة للأعمال والتفعيل والمستخدمين تمر عبر **`pg`** إلى **PostgreSQL** المحدد في `DATABASE_URL`. لا يوجد في الكود مسار «قاعدة محلية موازية» للأعمال.

## 5. التفعيل: بعيد أم محلي؟

### ما تثبته الشيفرة

- إدراج/تحديث المفاتيح: `server/src/services/activationService.ts` — جداول **`activation_keys`**, **`activation_events`**, و`system_settings` (`activation.status`).
- الجهاز: `activation_devices` عند توفر بصمة جهاز.
- الواجهة: `src/lib/api/activationApi.ts` تستدعي `/api/activation/*` — لا تخزين مفتاح خام في IndexedDB كمصدر حقيقة.

### استعلامات تحقق آمنة (يُنفَّذ على VPS)

```sql
-- حالة المفاتيح (بدون hash كامل)
SELECT status, key_suffix, plan_code, activation_count, updated_at
FROM activation_keys
ORDER BY updated_at DESC
LIMIT 5;

SELECT event_type, key_suffix, created_at
FROM activation_events
ORDER BY created_at DESC
LIMIT 10;

SELECT key, (value->>'active') AS active, (value->>'keySuffix') AS suffix_hint
FROM system_settings
WHERE key = 'activation.status'
LIMIT 5;
```

### المحلي

- العميل يعرض نتيجة **استجابة API** (`getActivationStatus`) — **نسخة عرض**، ليست قاعدة تفعيل.
- ملفات `server/generated/*.txt` بعد CLI التوليد: **نسخة مالك** فقط؛ التحقق السليم في القاعدة.

**الخلاصة:** **مصدر حقيقة التفعيل = PostgreSQL على خادم الـ API** (VPS إذا كان `DATABASE_URL` يشير إليه).

## 6. المستخدمون وتسجيل الدخول

### الشيفرة

- `server/src/routes/authRoutes.ts`: `SELECT ... FROM users WHERE username ...` ثم `bcrypt.compare`.
- كلمات المرور **ليست** نصاً صافٍ في القاعدة؛ التحقق من الهاش فقط.

### استعلام تحقق (آمن)

```sql
SELECT username, role, is_active, created_at
FROM users
WHERE username = 'admin';
-- لا تُعرَض password_hash كاملة في التقارير؛ يكفي وجود الصف وطول الهاش.
```

### المحلي

- **JWT** في `fabric_erp_api_token` (`client.ts`).
- لا يوجد تخزين قائمة مستخدمين أو كلمات مرور صافية للتحقق المحلي من الهوية.

**الخلاصة:** **المستخدمون والهاش = REMOTE (PostgreSQL)**؛ **الجلسة = LOCAL**.

## 7. بيانات الأعمال

- إنشاء مورد/عميل/صنف/تواب عبر API يترجم إلى `INSERT`/`UPDATE` في PostgreSQL في المسارات تحت `server/src/routes/` (مثل `supplierRoutes`, `fabricRollRoutes`, …).

### اختبار تحقق مقترح (لدى المالك)

1. إنشاء مورد باسم فريد: `REMOTE_TEST_SUPPLIER_<timestamp>` من الواجهة أو API.
2. على PostgreSQL:
   ```sql
   SELECT name, created_at FROM suppliers WHERE name LIKE 'REMOTE_TEST_SUPPLIER_%' ORDER BY created_at DESC LIMIT 3;
   ```
3. إعادة تشغيل العميل، تسجيل دخول من جديد — يجب أن يظهر السجل إذا كان الإدراج عبر API ناجحاً.

**ملاحظة:** صفحات تعتمد على **Zustand فقط** قد تعرض بيانات محلية قديمة؛ المصدر المعتمد للوحدات المتصلة بالـ API هو **الخادم**.

## 11. لا قاعدة أعمال محلية — تأكيد صريح

| السؤال | الجواب (من التبعيات + الشيفرة) |
|--------|----------------------------------|
| SQLite لمشروع التطبيق؟ | **لا** في `package.json` للتطبيق (لا `better-sqlite3` ولا محرك sqlite للأعمال). |
| IndexedDB لتخزين أعمال؟ | **لا** استخدام في `src/` لتخزين عملاء/أتواب كمصدر حقيقة. |
| JSON كمصدر حقيقة للأعمال؟ | **لا**؛ JSON في Electron للإعدادات فقط. |
| قاعدة Electron محلية للعملاء/الأتواب؟ | **لا**. تعليق صريح في `electron/main.ts`: PostgreSQL لا يُفتَح من Electron. |
| قاعدة تفعيل محلية منفصلة؟ | **لا**؛ التحقق عبر API والخادم يقرأ PostgreSQL. |

**جملة موجزة للمالك:**  
**لا توجد قاعدة أعمال محلية في مسار التشغيل الرسمي. مصدر الحقيقة للبيانات التشغيلية والتفعيل والمستخدمين هو PostgreSQL المتصل بخادم الـ API (`DATABASE_URL`).**  
إذا كان هذا الخادم على VPS و`DATABASE_URL` يشير إلى قاعدة على ذلك البيئة، فالبيانات **فعلياً** على البنية البعيدة — وليس «محلياً في العميل».

## 13. غموض / مخاطر

- **وحدات واجهة قديمة + Zustand:** قد تعرض حالة محلية قديمة؛ التصنيف يظل **REMOTE** للمصدر الحقيقي إذا كانت الصفحة تُحمَّل من API.
- **مسودة إعدادات في `localStorage`:** قد تختلف عن القاعدة حتى الحفظ — ليس تناقضاً مع «مصدر الحقيقة» بل تأخير مزامنة.
- **إثبات VPS بالاسم:** يتطلب مقارنة **مضيف** `DATABASE_URL` (على خادم API) مع عنوان VPS دون نشر السلسلة كاملة في الوثائق العامة.

## 14. الحكم النهائي

**المشروع يعتمد على PostgreSQL (عبر `DATABASE_URL` على خادم الـ API) كمصدر حقيقة للتفعيل والمستخدمين وبيانات الأعمال؛ التخزين المحلي للعميل محدود بالجلسة والإعدادات والتفضيلات.**  

لا يُظهر التدقيق وجود قاعدة أعمال SQLite أو بديل يخزّن العملاء/الأتواب محلياً كبديل عن PostgreSQL في مسار الإنتاج الموثق.
