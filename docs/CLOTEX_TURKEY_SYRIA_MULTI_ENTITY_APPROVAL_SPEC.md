# CLOTEX — مواصفة اعتماد: تركيا + سوريا (بانتظار موافقة الأب)

**الحالة:** مسودة معتمدة من فريق التطوير — **بانتظار الموافقة النهائية من العميل (الأب)**  
**التاريخ:** 2025-06-05  
**السياق:** نظام واحد + عزل بيانات. المعمل والمستودع في تركيا؛ مستودع في سوريا (ومعمل سوريا مستقبلاً).  
**التسليم السابق:** نظام + اشتراك سحابة بقيمة **2,000 USD** (النطاق الأساسي الحالي).

---

## 1. القرارات المعمارية المتفق عليها

### 1.1 نموذج التشغيل
- **نظام واحد** (CLOTEX) — وليس نظامين منفصلين.
- **كيانان مستقلان للبيانات:**
  - `CLOTEX-TR` — تركيا (معمل + مستودع)
  - `CLOTEX-SY` — سوريا (مستودع؛ معمل لاحقاً)
- **عزل البيانات** على مستوى الكيان (company) في قاعدة البيانات والـ API — وليس إخفاء قوائم في الواجهة فقط.

### 1.2 المستخدمون والصلاحيات
- كل مستخدم مربوط **بكيان واحد** (TR أو SY) — إلزامي.
- **عدة محاسبين لنفس الكيان** (مثلاً محاسبان في سوريا) — مسموح؛ نفس نطاق البيانات، اختلاف بالدور والصلاحيات فقط.
- **المدير الكبير (الأب):** صلاحية خاصة للوصول لأكثر من كيان + **مبدّل كيان** في الهيدر بعد الدخول (ليس في شاشة تسجيل الدخول للجميع).
- **الفرع داخل الكيان** (مثلاً دمشق / حلب): مرحلة لاحقة عند الحاجة — ليس شرط المرحلة الأولى.

### 1.3 تسجيل الدخول
- **لا** اختيار تركيا/سوريا في شاشة الدخول للموظف العادي.
- **لا** اكتشاف تلقائي بالموقع أو IP.
- **نعم:** تحديد الكيان تلقائياً من **حساب المستخدم** بعد المصادقة.
- للمدير فقط: تبديل الكيان من أعلى الشاشة بجانب مبدّل اللغة.

### 1.4 اللغات (i18n)
- زر أعلى الصفحة: **عربي ↔ تركي** (اتجاه RTL/LTR).
- الترجمة تشمل **واجهة النظام** (قوائم، أزرار، رسائل، تقارير) — وليس أسماء العملاء/الأصناف في قاعدة البيانات.
- **مرحلة 1 مقترحة:** شاشات التشغيل اليومية (دخول، لوحة، فواتير أساسية، مخزون).
- **مرحلة 2:** بقية الوحدات والتقارير والطباعة.

### 1.5 العملات
| الكيان | عملة افتراضية | ثانوية | ملاحظة |
|--------|---------------|--------|--------|
| CLOTEX-TR | **TRY** (ليرة تركية) | **USD** | التشغيل المحلي بالليرة؛ الدولار للعرض/التقارير/المشتريات الدولية لاحقاً |
| CLOTEX-SY | **USD** | (اختياري SYP لاحقاً) | كما هو الوضع الحالي |

- **العملة تُعرَّف على مستوى الكيان** في الإعدادات — المستخدم **يرث** عملة كيانه (لا يختار عملة أساسية لكل مستخدم على حدة).
- **سعر صرف** يدوي/دوري للتحويل والعرض (قرار تشغيلي للأب).

### 1.6 مناقلة المخزون تركيا → سوريا
- قسم في **المخزون:** مناقلة بين مستودع تركيا ومستودع سوريا (عملية **بين كيانين**).
- حالات مقترحة: مسودة → معلّقة/في الطريق → مكتملة (أو ملغاة).
- **تأكيد بكلمة مرور الطرف الآخر:** **مؤجّل** — المدير الكبير ينفذ العمليات حالياً من طرف واحد.
- سياسة المخزون عند الإرسال: حجز أو خصم مع حالة «في الطريق» (يُحسم عند التنفيذ).

### 1.7 شاشات الإدارة المطلوب تطويرها
**إعدادات الكيان:**
- اسم الكيان، العملة الأساسية، العملة الثانوية، سعر الصرف.

**إضافة/تعديل مستخدم:**
- الكيان (TR / SY) — إلزامي
- الدور والصلاحيات
- عرض عملة الكيان للمعلومة
- (لاحقاً) فرع/مستودع محدد داخل الكيان
- للمدير: وصول متعدد الكيانات

**الهيدر بعد الدخول:**
- مبدّل لغة (عربي / Türkçe)
- مبدّل كيان (للمدير فقط)

## 2. ما هو خارج نطاق هذه المواصفة (يحتاج عرض منفصل)

### 2.1 قسم التصنيع
- الوضع الحالي في الكود: **واجهة تجريبية (Mock)** بدون API أو قاعدة بيانات.
- تطوير تصنيع حقيقي لمعمل أقمشة يتضمن (حسب الاتفاق): BoM، أوامر تصنيع، استهلاك بكرات/أمتار، هالك، مراحل إنتاج، تكلفة — **مشروع كبير منفصل** عن عزل تركيا/سوريا.

---

## 3. أسئلة مفتوحة لموافقة الأب

1. هل المدير يريد **تبديل كيان** بعد الدخول أم حساباً افتراضياً لتركيا فقط؟
2. هل في تركيا يوجد **صندوق/مشتريات بالدولار** من البداية أم **عرض USD** فقط مع تسجيل TRY؟
3. أولوية اللغة التركية: **شاشات أساسية** أولاً أم **النظام كاملاً** دفعة واحدة؟
4. نطاق التصنيع المطلوب (مراحل، تقارير، ربط مخزون البكرات) — ورشة عمل منفصلة.

---

## 4. تسلسل تنفيذ مقترح (بعد الموافقة)

```
1) تعريف الكيانين TR + SY وإعداد العملات
2) ربط المستخدمين بالكيان + صلاحية المدير متعدد الكيانات
3) مراجعة API: فلترة إلزامية بـ company_id
4) واجهة المستخدمين وإعدادات الكيان
5) مناقلة مخزون بين الكيانين (بدون كلمة مرور طرف ثانٍ — مرحلة 1)
6) i18n تركي — حسب النطاق المعتمد
7) (منفصل) التصنيع الحقيقي
```
