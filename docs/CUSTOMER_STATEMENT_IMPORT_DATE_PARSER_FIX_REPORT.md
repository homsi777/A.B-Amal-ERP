# تقرير إصلاح قراءة تواريخ Excel في استيراد كشف حساب العميل

## 1. Executive Summary

تم تنفيذ إصلاح محدود وآمن لمسار استيراد كشف حساب العميل من Excel بدون تغيير منطق المحاسبة أو طريقة إنشاء الفواتير أو السندات أو القيود.

الهدف كان منع الخطر الأساسي: عند فشل قراءة التاريخ كان النظام يستخدم تاريخ اليوم بصمت، وهذا قد يجعل سند قبض أو قيد محاسبي بتاريخ خاطئ. الآن أصبحت قراءة التاريخ تعطي نتيجة صريحة: تاريخ صحيح موحد بصيغة `YYYY-MM-DD` أو خطأ يمنع الاعتماد.

## 2. What Was Wrong Before

المشكلة السابقة كانت في `src/components/customers/CustomerStatementImportModal.tsx`:

- دالة قراءة التاريخ القديمة كانت تدعم نمطاً محدوداً فقط مثل `dd/mm/yyyy`.
- إذا لم تجد تاريخاً مطابقاً كانت ترجع تاريخ اليوم:
  `new Date().toISOString().slice(0, 10)`
- تواريخ Excel الرقمية مثل `46142` لم تكن تقرأ كتاريخ.
- دفعات Excel التي تحمل تاريخاً رقمياً داخلياً كان يمكن تجاهلها أو تحويلها بشكل خاطئ.
- Backend كان يقبل نص التاريخ بعد التحقق العام من Zod بدون تحقق date-only صارم.

هذا خطر لأن دفعات الكشف المستوردة تنشئ سندات قبض مؤكدة وحركات صندوق وقيود مرتبطة عبر المسار الحالي.

## 4. Date Formats Now Supported

تمت إضافة parser مستقل يدعم:

- أرقام Excel التسلسلية مثل `46142`
- كائنات `Date` إذا أرجعتها مكتبة xlsx
- `dd/mm/yyyy`
- `dd-mm-yyyy`
- `dd.mm.yyyy`
- `d/m/yyyy`
- `yyyy-mm-dd`
- `yyyy/mm/dd`
- `mm/dd/yy` عندما يكون واضحاً أن الجزء الثاني يوم، مثل `4/30/26`
- نص يحتوي كلمة `مرتجع` مع تاريخ مضمّن، مثل `مرتجع 4/30/26`
- الأرقام العربية والهندية داخل النص

قاعدة الالتباس الحالية:

- إذا كان التاريخ مثل `7\4\2026` وكلا الرقمين يصلحان كشهر/يوم، يتم تفسيره حسب نمط العمل العربي `day/month/year`.
- لذلك `7\4\2026` يصبح `2026-04-07`.

- `تم اكتشاف تاريخ Excel رقمي وتم تحويله إلى: 2026-04-30`

## 6. What Happens On Invalid Dates

لم يعد النظام يرجع تاريخ اليوم عند الفشل.

عند تاريخ غير صالح مثل:

- فارغ
- `35/14/2026`
- نص عشوائي

يرجع parser نتيجة فشل، وتظهر رسالة في المعاينة، ويتم تعطيل زر تأكيد الاستيراد عند وجود أخطاء تواريخ ملزمة.

الأمثلة التي تمنع الاعتماد:

- تاريخ كشف غير مفهوم
- تاريخ بند مبيعات غير مفهوم
- تاريخ دفعة أو مرتجع غير مفهوم

## 7. Frontend Preview Behavior

في شاشة استيراد كشف العميل:

- تعرض المعاينة التاريخ normalized بصيغة `YYYY-MM-DD`.
- تعرض أصل التاريخ في جدول بنود المبيعات.
- تعرض مصدر التاريخ في جدول الدفعات والمرتجعات.
- تظهر كتلة تحذير حمراء عند وجود تواريخ يجب مراجعتها.
- يتم تعطيل زر `تأكيد الاستيراد` إذا وجدت أخطاء تواريخ.

لم يتم تغيير تصميم الشاشة أو تدفقها الأساسي.

## 8. Backend Validation Behavior

في `POST /api/customers/import-statement` تمت إضافة تحقق date-only:

- `orderDate` يجب أن يكون `YYYY-MM-DD` صالحاً.
- كل `saleLines[].date` يجب أن يكون `YYYY-MM-DD` صالحاً.
- كل `payments[].date` و `returnPayments[].date` يجب أن يكون `YYYY-MM-DD` صالحاً.

إذا وصل payload غير صالح، يرجع backend خطأ validation ولا يسمح للتاريخ بالوصول إلى SQL أو إنشاء سندات/قيود.

## 9. Original Date Preservation Behavior

بدون إضافة جداول أو أعمدة جديدة:

- تم تمرير `originalDateValue` و `dateParseSource` من الواجهة إلى backend.
- تم حفظهما داخل `metadata` لبنود فواتير المبيعات المستوردة.
- تمت إضافتهما إلى ملاحظات سندات القبض المستوردة بصيغة نصية.
- تمت إضافتهما إلى وصف قيود المرتجع/الحسم المستوردة.

هذا يعطي أثر تدقيق عملي دون تغيير schema.

## 13. Remaining Limitations

- لم يتم تغيير التصنيف المحاسبي للسطور.
- لم تتم إضافة staging tables.
- لم يتم تعديل طريقة إنشاء الفواتير أو سندات القبض أو القيود.
- لم يتم دعم supplier import.
- ما زال تحديد ما إذا كان السطر رصيد افتتاحي أو فاتورة قديمة أو دفعة أو مرتجع يحتاج مرحلة محاسبية مستقلة.
- عند التواريخ الملتبسة جداً مثل `7/4/2026` تم اعتماد قاعدة `day/month/year` لأنها أنسب للملفات العربية الحالية، لكن أي ملف أمريكي بالكامل يحتاج سياسة واضحة قبل الاستيراد.
