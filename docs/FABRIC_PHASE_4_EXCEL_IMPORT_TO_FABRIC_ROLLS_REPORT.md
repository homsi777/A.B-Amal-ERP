## 1. الملخص التنفيذي

تم في هذه المرحلة بناء محرك استيراد Excel كامل واحترافي يتيح لمدير المستودع رفع ملف Excel الخاص بفاتورة الشراء أو بيان الحاوية، مراجعة البيانات بشكل مرئي، وتأكيد الاستيراد لإضافة الأتواب مباشرةً إلى قاعدة البيانات PostgreSQL.

**المبدأ الأساسي:** لا يُضاف أي ثوب إلى `fabric_rolls` قبل تأكيد المستخدم صراحةً.

### إنجازات المرحلة:
- 🗄️ هجرة قاعدة بيانات جديدة: جدولان لتتبع دفعات الاستيراد وصفوفها
- 🔧 6 نقاط API backend متكاملة للمعاينة والتأكيد والإلغاء والتاريخ
- 🧠 كاشف أعمدة ذكي يدعم العربية والتركية والإنجليزية (أكثر من 120 اسم مستعار)
- ✅ نظام تحقق شامل: أخطاء صارمة وتحذيرات قابلة للتجاوز
- 🔗 مطابقة تلقائية للخامات والألوان والمتغيرات مع خيار الإنشاء التلقائي
- 💻 واجهة مستخدم 3-خطوات بالعربية RTL
- 📋 صفحة تاريخ دفعات الاستيراد
- 🧪 33 اختبار API كل النتائج ناجحة

## 5. نقاط API الـ Backend

### `POST /api/purchases/import/preview`

رفع بيانات Excel كـ JSON وإنشاء دفعة معاينة.

**المدخلات:**
```json
{
  "fileName": "filename.xlsx",
  "sheetName": "Sheet1",
  "warehouseId": "uuid",
  "supplierId": "uuid (اختياري)",
  "importMode": "MATCH_ONLY | CREATE_MISSING_MASTER_DATA",
  "headers": ["اسم الخامة", "الطول", ...],
  "rows": [[row1_values], [row2_values], ...]
}
```

**الإخراج:** ملخص الدفعة مع `batchId` وعدد الصفوف وإحصاءات الأعمدة المكتشفة.

**القيود:** لا يُنشئ أي ثوب في هذه المرحلة.

---

### `GET /api/purchases/import/:id`

جلب تفاصيل الدفعة (status, counts, totals).

---

### `GET /api/purchases/import/:id/rows`

جلب صفوف الدفعة مع الفلترة حسب الحالة وصفحات البيانات.

**Query params:** `status`, `page`, `pageSize`

---

### `POST /api/purchases/import/:id/confirm`

تأكيد الاستيراد وإنشاء الأتواب. **تشغيل في transaction كاملة.**

**المدخلات:** `{ "allowWarnings": true/false }`

**الأمان:**
- يرفض إذا كانت الدفعة `CONFIRMED` مسبقاً → 409
- يرفض إذا كانت `CANCELLED` → 400
- يرفض إذا يوجد صفوف أخطاء → 400
- يرفض إذا يوجد تحذيرات ولم يُرسَل `allowWarnings=true` → 400

---

### `POST /api/purchases/import/:id/cancel`

إلغاء دفعة في حالة PREVIEW. لا يمكن إلغاء دفعة CONFIRMED.

---

### `GET /api/purchases/import`

قائمة دفعات الاستيراد السابقة مع ترقيم الصفحات.

## 7. قواعد التحقق من البيانات

### أخطاء صارمة (تمنع الاستيراد):

| الخطأ | الرسالة |
|-------|---------|
| اسم/كود الخامة مفقود | اسم الخامة أو كودها مطلوب |
| طول غير صالح | الطول غير صالح — يجب أن يكون رقماً موجباً |
| عرض غير صالح | العرض يجب أن يكون أكبر من صفر |
| GSM غير صالح | GSM يجب أن يكون أكبر من صفر |
| باركود مكرر في قاعدة البيانات | باركود موجود مسبقاً في قاعدة البيانات |
| باركود مكرر داخل الملف | باركود مكرر داخل الملف |
| خامة غير موجودة (وضع MATCH_ONLY) | الخامة غير موجودة في النظام |

### تحذيرات (قابلة للتجاوز):

| التحذير | الإجراء |
|---------|---------|
| باركود مفقود | سيُولَّد تلقائياً عند التأكيد |
| لون مفقود | استيراد بدون لون |
| طول مفقود | تسجيل بصفر |
| سيُنشأ عنصر/لون/متغير جديد | تحذير للإعلام |
| فرق الوزن > 10% | فرق الوزن الفعلي عن المحسوب أكبر من 10% |
| رقم الثوب مفقود | تحذير فقط |
| رقم الحاوية مفقود | تحذير فقط |
| رقم فاتورة الشراء مفقود | تحذير فقط |

### قاعدة فرق الوزن:
```
variancePercent = abs(actualKg - calculatedKg) / calculatedKg * 100
إذا variance > 10% → تحذير
```

## 8. قواعد المطابقة وإنشاء البيانات الرئيسية

### مطابقة الخامة (بالأولوية):
1. `internalMaterialCode` ← `fabric_items.internal_code`
2. `supplierMaterialCode` ← `fabric_items.supplier_code`
3. `materialName` ← `fabric_items.name` (مطابقة نصية)
4. إذا لم توجد + `CREATE_MISSING_MASTER_DATA` → إنشاء خامة جديدة

### مطابقة اللون (بالأولوية):
1. `colorCode` ← `fabric_colors.color_code`
2. `supplierColorCode` ← `fabric_colors.supplier_color_code`
3. `colorName` ← `fabric_colors.name_ar`
4. `colorNameTr` ← `fabric_colors.name_tr`
5. إذا لم يوجد + `CREATE_MISSING_MASTER_DATA` → إنشاء لون جديد

### مطابقة المتغير (بالأولوية):
1. `item_id + color_id + width_cm + gsm` مطابقة تامة
2. إذا لم يوجد + `CREATE_MISSING_MASTER_DATA` → إنشاء متغير جديد

**ملاحظة مهمة:** قاعدة `fabric_colors` لا تحتوي على unique constraint، لذلك تم استخدام SELECT + INSERT بدلاً من ON CONFLICT.

## 9. سلوك تأكيد الاستيراد

عند تأكيد دفعة:

1. **التحقق من الحالة:** رفض إذا CONFIRMED/CANCELLED أو يوجد أخطاء
2. **تشغيل Transaction:** كل العمليات في transaction واحدة — rollback كامل عند أي فشل
3. **إنشاء Master Data** (إذا `CREATE_MISSING_MASTER_DATA`): خامات/ألوان/متغيرات مفقودة
4. **توليد Barcode:** إذا لم يُحدَّد باركود في الصف → `generateBarcode()` من Phase 3
5. **حساب الوزن:** `calcWeight(length, width, gsm)` من Phase 3
6. **إنشاء `fabric_roll`** في قاعدة البيانات
7. **إنشاء `inventory_movement`** بنوع `PURCHASE_RECEIPT` و `reference_type = IMPORT_BATCH`
8. **تحديث الصف** بـ `status = IMPORTED` و `created_roll_id`
9. **تحديث الدفعة** بـ `status = CONFIRMED` والإحصاءات الإجمالية
10. **Commit Transaction**

**قرار معماري:** عند فشل أي صف → rollback كامل للـ transaction لتجنب الاستيراد الجزئي المفاجئ.
